A* Pathfinding Algorithm

A* finds the shortest path between two points by balancing actual travel cost and a heuristic estimate of remaining distance.
Author

Andreas Bogossian

Published

March 19, 2026

How does Google Maps decide which route to show you — not just a route, but the best one — in under a second, across a road network with millions of edges?

The problem with brute force

Imagine you’re standing at one corner of a city grid, trying to reach a destination on the opposite side. The naive approach is to explore every possible path until you find the shortest one. That works fine for a 5×5 grid. But the number of paths grows as O(b^d), where b is the branching factor (how many directions you can move at each step) and d is the depth (how many steps to the goal). On a real road network, this becomes completely intractable.

Dijkstra’s algorithm improves on brute force by always expanding the cheapest node seen so far — but it still fans out in every direction equally, spending effort on roads that are taking you further from your destination.

What you really want is an algorithm that focuses its search toward the goal. That’s the core idea behind A*.

The A* cost function

Every node n in A* is assigned a score:

f(n) = g(n) + h(n)

  • g(n) is the exact cost of the path from the start to n — we’ve already walked this, so we know it precisely.
  • h(n) is the estimated cost from n to the goal — this is a heuristic, an educated guess.

At each step, A* pops the node with the lowest f(n) from a priority queue (the open set) and expands it. Once a node is fully evaluated, it moves to the closed set and won’t be revisited.

The elegance is in the balance. g(n) keeps the algorithm honest — it won’t ignore a cheap path it’s already found. h(n) keeps the algorithm focused — it steers effort toward the goal instead of spreading uniformly outward.

Admissibility and optimality

The guarantee that A* finds the optimal path rests on one condition: the heuristic must be admissible, meaning it never overestimates the true remaining cost.

h(n) \leq h^*(n)

where h^*(n) is the true cost from n to the goal. If this holds, A* is guaranteed to find the shortest path. Overestimate, and you risk the algorithm committing to a path that looks cheap but isn’t.

Two special cases are worth internalising:

  • When h(n) = 0 for all nodes, A* degenerates into Dijkstra’s algorithm — it explores uniformly in every direction, ignoring the goal entirely.
  • When h(n) exactly equals h^*(n), A* expands only nodes that lie on an optimal path — a theoretical ideal.

In practice, a good heuristic sits somewhere in between: cheap to compute, never overestimates, and close enough to the truth to make the search directional.

Choosing a heuristic

The right heuristic depends on the movement model of your problem:

Heuristic Formula Best for
Manhattan \|x_1 - x_2\| + \|y_1 - y_2\| 4-directional grids
Euclidean \sqrt{(x_1-x_2)^2 + (y_1-y_2)^2} any-angle movement
Chebyshev \max(\|x_1-x_2\|, \|y_1-y_2\|) 8-directional grids

On a 4-directional grid like the demo below, Manhattan distance is exact — you can’t move diagonally, so it never overestimates. Euclidean distance will underestimate on such grids (straight-line distance is shorter than any grid path), so it’s admissible but less informed. Chebyshev distance assumes diagonal moves are free, which would overestimate on a 4-directional grid — inadmissible, so don’t use it here.

Weighted A*

A useful variant inflates the heuristic by a weight w > 1:

f(n) = g(n) + w \cdot h(n)

This makes the search greedier — it chases the goal more aggressively, exploring fewer nodes. The trade-off is that the optimality guarantee weakens: the path found is at most w times longer than the optimal path. At w = 1 you get standard A*; as w grows the algorithm approaches a pure greedy best-first search. Try the weight slider in the demo to see this effect directly.

So what does this actually mean?

A* is essentially a guided Dijkstra. It adds a compass to the search — the heuristic — that tells it “the goal is roughly that way.” The open set is the frontier of what’s been discovered but not yet committed to. The closed set is what’s been fully explored. At each tick, the algorithm asks: of everything on the frontier, what has the best combination of how far I’ve come and how far I still need to go? That node goes next.

The demo below makes this tangible. The blue-green cells are the open set (discovered, not yet expanded). The dark green cells are the closed set (expanded). The yellow path is the solution once the algorithm reaches the end node. Try switching heuristics and watch how the search shape changes — Dijkstra (zero) fans out evenly like a ripple, while Manhattan stays tightly focused. Then push the weight slider above 1 and watch the search narrow further at the cost of path optimality.

Discussion

A* is impressively general. The same framework that navigates a 2D grid also powers route planning in GPS systems, NPC pathfinding in video games, puzzle solvers (the 15-puzzle, Rubik’s cube with appropriate state representations), and network routing. Anywhere you can define a cost function and an admissible heuristic, A* applies.

That said, it has real limitations. The open set is held in memory, and in the worst case it can grow to hold every reachable node — a problem on very large graphs. On a grid with 10^6 cells and a bad heuristic, memory becomes the bottleneck before time does. Variants like IDA* (iterative deepening A*) trade the memory guarantee for a depth-first search pattern, keeping memory usage linear at the cost of re-expanding nodes.

The other limitation is more subtle: A* is only as good as its heuristic. A weak heuristic (like zero) wastes time exploring irrelevant nodes. A heuristic that inadvertently overestimates silently discards optimal paths. Designing a good heuristic for a novel problem is often the real work — the algorithm itself is just infrastructure.

One other thing to notice in the demo: on a maze with many dead ends, even Manhattan distance explores a surprisingly large fraction of the grid. The heuristic can’t see through walls. This is a fundamental constraint — the algorithm has no map of the maze structure, only a straight-line estimate of remaining distance. Algorithms that precompute structure (like JPS, Jump Point Search, or hierarchical pathfinders) can do significantly better on structured environments.

Summary

A* solves the shortest path problem by combining two pieces of information at every step: the exact cost of how you got here (g), and a heuristic estimate of how far you still have to go (h). This sum, f = g + h, serves as a priority score. By always expanding the node with the lowest f, the algorithm balances completeness (it won’t miss the optimal path, provided the heuristic is admissible) with focus (it avoids wasting effort on nodes that are clearly far from good).

The insight is simple: if you want to search smarter, not harder, give the algorithm a compass. The heuristic is that compass — and the quality of your heuristic determines how straight the algorithm walks toward the goal.

If this sparked your interest, a natural next step is Dijkstra’s algorithm (A* without the compass) or Jump Point Search, which exploits grid symmetry to skip large swathes of the search space entirely.
